Maintainable JavaScript

by Peter, Jeremy, and Nicole

Why Use a Framework?

  • asynchronous page load
  • code management with JQuery sucks
  • help with well-defined application architecture
  • responsive UI
  • well-structured and maintainable code

Important Terms

2-way data binding Dependency Injection Routing Templates Boilerplate (code) Server-Side Rendering String-based templates/DOM-based templates

Angular JS

  1. Size: 36kB
  2. 2 way data binding
  3. Declarative DOM-based View Template
  4. Reusable components aka directives
  5. Clear routing
  6. Can create custom DOM elements
  7. Has it's own concepts - e.g. scopes, directives, transclusion
  8. No server side rendering
  9. Transitioning from 1.0 to 2.0

Backbone JS

  1. Size: 6.3kB
  2. No 2 way data binding
  3. String-based View Template: usually underscore.js or handlebars.js
  4. Use a mixin system for code reuse
  5. Views manipulate the DOM directly
  6. No server side rendering
  7. Barebones, allowed to mix and match to optimize functionality
  8. Can be used for only parts of the page

React JS

  1. Size: 120kB
  2. Unidirectional data flow
  3. No routing
  4. Use a mixin system for code reuse
  5. Very fast
  6. Easy to learn
  7. Can be used to create mobile apps
  8. supports server side rendering
  9. Takes care of the V in MVC
  10. No templates, components generate the UI so HTML lives in the JS

Ember JS

  1. Size: 95kB
  2. 2 way data binding
  3. String-based view template - handlebars.js
  4. Reusable components at the view level but not supported at controller level
  5. Slick routing
  6. JQuery and Handlebars are required dependencies
  7. Everything is built in - like rails convention over config
  8. Templating library, routing, etc. included
  9. More of a complete solution
  10. Steep learning curve because there's so much included
  11. No server side rendering
  12. Controls the entire page
  13. Sucks to set up with Rails
